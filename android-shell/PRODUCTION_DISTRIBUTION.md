# SpeyPOS Production Distribution Strategy

This document outlines the strategy for distributing the SpeyPOS Android Shell app to acquired customers via a secure, self-hosted mechanism.

## 1. Infrastructure Architecture

### AWS S3 Storage
- **Bucket**: `speypos-distribution-private` (or similar).
- **Access Control**: 
    - Public access: **Blocked**.
    - Identity & Access Management (IAM): A specific IAM user for GitHub Actions with `s3:PutObject` permissions.
    - Retrieval: Only via **Signed URLs** generated by the SpeyPOS Cloud Backend.

### Storage Layout
```text
/release/
  /latest/
    speypos-shell-latest.apk
    update.json
  /v1.0.5/
    speypos-shell-v1.0.5.apk
```

### `update.json` Schema
```json
{
  "versionCode": 105,
  "versionName": "1.0.5",
  "minSdkVersion": 26,
  "priority": "HIGH",
  "releaseNotes": "Security updates and thermal printer fixes.",
  "timestamp": "2024-05-20T10:00:00Z"
}
```

---

## 2. CI/CD Pipeline (GitHub Actions)

When a tag is pushed (e.g., `v1.0.5`) or a release is created in GitHub:

1.  **Build**: Assemble the release APK (`./gradlew assembleRelease`).
2.  **Sign**: Sign the APK using the production Keystore.
3.  **Metadata**: Generate the `update.json` based on the build outputs.
4.  **Upload**:
    - Upload the versioned APK to `s3://.../release/v1.0.5/`.
    - Overwrite `s3://.../release/latest/speypos-shell-latest.apk`.
    - Overwrite `s3://.../release/latest/update.json`.
5.  **Invalidate Cache**: If CloudFront is used, invalidate the `latest/` path.

---

## 3. Installation Workflows

### First-Time Installation (Technician)
1.  Technician logs into the SpeyPOS Admin Portal.
2.  Portal generates a **One-Time Presigned URL** valid for 30 minutes.
3.  Technician scans a QR code on the device or enters the URL in the mobile browser.
4.  Device downloads and installs the initial shell.

### Self-Update Mechanism (App Logic)
1.  **Check Phase**: Every 24 hours (or on App Start), the app calls the SpeyPOS Cloud Backend: `GET /api/v1/system/update-check`.
2.  **Verification**: 
    - The Backend fetches `update.json` from S3.
    - Compare `versionCode`.
    - If a newer version exists, the Backend returns the metadata + a **Short-lived Presigned URL** for the APK.
3.  **Download Phase**: The app downloads the APK to its internal cache.
4.  **Install Phase**:
    - The app checks if it has `REQUEST_INSTALL_PACKAGES` permission.
    - Prompts the user (technician/staff) to install the update.
    - Uses `FileProvider` to launch the Package Installer.

---

## 4. Security & Compliance

- **Authentication**: Update checks are authenticated using the Device API Key (linked during the handshake process).
- **Integrity**: The `update.json` includes a SHA-256 hash of the APK. The app verifies the hash after download before attempting installation.
- **Rollback**: If a critical bug is found, `release/latest/` can be manually pointed back to a previous stable version in S3.

---

## 5. Required Implementation Steps (Android)

1.  **Permissions**: Add `REQUEST_INSTALL_PACKAGES`.
2.  **FileProvider**: Configure a `FileProvider` in `AndroidManifest.xml` to share the downloaded APK with the system installer.
3.  **UpdateService**: Create a Kotlin service/manager to handle the polling, downloading, and hash verification.
4.  **UI Interaction**: Add a non-intrusive update progress indicator in the native shell (or a modal if the update is `HIGH` priority).
